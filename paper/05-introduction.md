In materials characterization, spectrum analysis is the step that turns a measurement into evidence. An X-ray diffraction (XRD) pattern becomes a phase identification only after peak finding under instrument- and sample-dependent backgrounds, indexing against reference databases such as the Materials Project[8] or ICSD-derived resources, and, where appropriate, whole-pattern fitting or Rietveld-style refinement[9] using established tools such as GSAS-II[10] or BGMN[11]. An X-ray photoelectron spectroscopy (XPS) survey becomes a chemical-state assignment only after Shirley[12] background subtraction, charge correction against an internal reference, peak deconvolution under chemically meaningful constraints, and quantification using Scofield[13] cross sections. A Raman spectrum becomes a structural assignment only after baseline correction and matching against curated libraries such as RRUFF[14]. These are executable procedures, not prose, and each carries a body of community convention that is easy to read about and hard to apply correctly the first time.

The result is an expertise barrier that is real for working researchers and steeper for everyone else. A graduate student running their first XPS fit, a chemist who occasionally needs an XRD interpretation outside their main expertise, and a synthesis lab that has to characterize many samples per week all face the same friction: choose the right tool, set parameters in domain-correct ways, and combine the outputs into a defensible interpretation. Even for specialists, the per-sample wall time — open vendor software, run a fitting script, search the literature, copy results into a report — accumulates quickly. Materials spectrum analysis is therefore not only a domain-knowledge bottleneck but a time bottleneck.

Recent AI-for-materials systems offer an attractive direction. Coscientist[3] couples GPT-4 with autonomous reaction planning; ChemCrow[4] augments an LLM with chemistry tools for synthesis planning; A-Lab[5] drives robotic synthesis from agent-issued plans. These systems frame the ambition as autonomous discovery and are important demonstrations of what agentic AI may eventually automate. But for most users facing a stack of unanalysed spectra, autonomous discovery is not the immediate need. The more immediate, broadly useful value of an LLM agent is to lower the expertise barrier and shorten the time-to-interpretable-result in spectrum analysis itself. ReAct[1] and Toolformer[2] showed that interleaving reasoning with tool invocation can ground model outputs in external computation rather than parametric memory; the open question is how to make that abstraction usable in the day-to-day, human-in-the-loop workflow of materials spectrum interpretation.

Plain chat is not enough. When a user asks an LLM to interpret an XRD pattern or fit an XPS spectrum, the model can emit 2θ positions, peak widths, binding energies, or stoichiometric ratios in confident prose. Without an evidence trail, the user cannot tell which tool, if any, produced each number; what intermediate state was passed into the calculation; whether a low-confidence match was ignored; or how to reproduce the answer months later. Recent surveys of hallucination in language models[6,7] make clear why this is risky: generated and grounded claims can appear equally fluent. In spectrum analysis specifically, a wrong number does not stay in the chat: it propagates into reports, into supervisor reviews, and into decisions about the next batch of samples.

Current software environments compound the problem. A single spectrum interpretation may require vendor software for ingestion, Python or MATLAB scripts for fitting, web portals for literature search, reference managers for citations, and a separate editor for the report. Provenance, when preserved at all, often lives in screenshots, exported CSV files, and ad-hoc filenames. This fragmentation is already a known obstacle to reproducible computational science[15,16]. Adding an LLM on top does not automatically solve it: the model cannot see the fitting script, the spectrum viewer cannot see the literature context, and the report editor cannot see the tool calls that produced the conclusion. What is missing is not another autonomous agent in isolation, but a workspace in which the agent, the analysis tools, the artifacts, the literature evidence, and the human reviewer share one auditable surface — so that the expertise barrier is partially absorbed into the workspace itself.

We present **Lattice**, a tool-grounded scientific agent workspace designed to lower the expertise barrier and shorten the time-to-interpretable-result in materials spectrum analysis. Lattice places an LLM agent, a catalog of characterization tools, a Python scientific worker, a literature-and-RAG subsystem, a typed artifact workspace, and approval gates inside a single Electron desktop application that runs on the user's machine. A user describes a spectrum and a question in natural language; the agent invokes the appropriate tools — peak detection, charge correction, library search, profile fitting, literature retrieval — and each tool produces a typed artifact (spectrum, peak table, fit result, candidate-phase list, paper, report) that is rendered in the workspace, persisted across sessions, and addressable by later tool calls. Approval gates intercept high-risk or human-in-the-loop steps such as host-side compute execution and editable XPS peak deconvolutions, so the user reviews intermediate state rather than only a final answer. Lattice captures every tool step, input, output, artifact, and final response during execution, so the evidence path is inspectable rather than implicit. The design reframes AI's role: not autonomous materials discoverer, but a workspace that makes a defensible spectrum interpretation accessible to non-specialists and faster to produce for specialists.

This paper makes three contributions.

**C1 — System.** We describe the architecture of Lattice: a multi-turn agent orchestrator with permission modes and approval gates, a tool catalog spanning spectroscopy, compute, literature, and workspace operations, a JSON-RPC Python worker for scientific computation, and an artifact-centered workspace persisted across sessions.

**C2 — Workflows.** We walk through four representative materials-characterization workflows — XRD phase identification with approximate whole-pattern fitting, XPS charge correction and peak deconvolution, Raman database identification, and literature-grounded interpretation — and show, at the level of individual tool calls and artifacts, how Lattice turns each from a chat exchange or a multi-tool patchwork into a single auditable analysis path.

**C3 — Compact evaluation.** We report a small but controlled comparison: twelve representative tasks, one tool-capable model held constant, two repeats per condition, run under Lattice and under a same-model chat-only baseline. We measure tool-grounded answer rate, unsupported numeric claims, artifact reproducibility, final-answer correctness, interpretation quality, expertise required (proxied by intervention and edit count), and time-to-result. The evaluation is intentionally compact; its purpose is to test whether the workspace lowers the expertise and time cost of producing a defensible interpretation, not to claim a benchmark ranking.

The remainder of the paper is organised as follows. Section 2 details the system design. Section 3 walks through the four characterization workflows. Section 4 specifies the evaluation protocol. Section 5 reports results and case studies. Section 6 discusses limits and failure modes, and Section 7 concludes.
