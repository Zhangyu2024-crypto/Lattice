# C1 figure notes

> A single connected paragraph describing the four C1 system figures in `figures/`. The images themselves no longer embed `Figure 1/2/3/4` numbering or captions; numbers and captions are controlled by the main paper text.

The four figures are arranged in a "whole system → one execution → safety policy → evidence trail" progression that visualises C1 in four layers. `fig-c1-architecture` gives the overall picture: the scientist and the LLM provider sit at the edges, while an agent orchestrator, a tool catalog of roughly seventy LocalTools, a Python scientific worker (JSON-RPC over stdio, around twenty-two methods, bundled offline XRD/XPS/Raman/RAG/CIF databases), an artifact-centered canvas (about thirty typed artifact kinds, Pro workbenches, an evidence DAG, IndexedDB persistence) and a two-layer approval surface (pre-exec gating on trust level plus permission mode, post-exec proposal cards) all live inside one local desktop workspace; the point is that Lattice's contribution is not any single component but the composition of these five subsystems into one human-in-the-loop loop. `fig-c1-agent-loop` collapses that architecture into a single bounded agent turn: the scientist's question enters the orchestrator, which builds a prompt from the transcript plus a context-filtered tool schema and dispatches the LLM call; the LLM emits a `tool_use`, the tool executes and patches the active xrd-pro workbench artifact, the result returns with a `sourceStepId` and is folded back as a `tool_result` for the next LLM call, possibly triggering a follow-up `xrd_search_phases`, until the model returns a plain-text reply — backed by an abort signal, an iteration ceiling, and a loop detector that terminates when the canonicalised tool-call signature is unchanged for three consecutive rounds — which is what makes the final answer tool-grounded rather than free-form chat. `fig-c1-approval-matrix` externalises the safety policy as a two-dimensional table: the columns are the tool's intrinsic trust level (safe / sandboxed / localWrite / hostExec / review-edit cards) and the rows are the session's current permission mode (normal / auto-accept / read-only / yolo), with every cell labelled auto, ask, or deny; the message is that safety is not hard-coded into individual tool implementations but emerges from the cross-product of "tool risk × session policy", so the same tool can run conservatively (ask) on a sensitive sample and aggressively (auto) on a parameter sweep without code changes. `fig-c1-artifact-dag` closes the sequence by tracing the evidence graph of one XRD interpretation: a raw spectrum enters the xrd-pro workbench, `detect_peaks` and `xrd_search_phases` branch off a peak table and a candidate-phase list, both feed into `xrd_refine`'s approximate pseudo-Voigt fit, which then fans out to a `plot_spectrum` artifact and a `paper_rag_ask` literature context, all converging on a final answer whose every numeric claim carries `sourceStepId` references back to the tool step that produced it; edges are parent pointers, the workbench node accumulates incremental patches, and every node persists across sessions, so intermediate evidence is never lost in chat text. Together the four panels answer C1's single thesis: **Lattice's contribution is not a smarter answer but an auditable workspace whose cost of staying in the loop is low enough that a scientist will actually do so**.
